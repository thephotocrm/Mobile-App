1. Situation

You’ve built a stateful AI wizard that:

Maintains conversation memory

Forces strict JSON output

Performs slot extraction + dialogue + readiness gating

Uses GPT-5 with chat.completions.create

Relies on perfect compliance for downstream automation

That is a fragile setup unless the model + API usage are aligned exactly right.

Right now, they aren’t.

2. Assumptions / Constraints (Important)

You’re using the OpenAI Node SDK

You’re calling openai.chat.completions.create

You expect strict JSON every time

You’re mixing conversation logic + extraction + control signals in one response

You’re deploying via Replit / Railway (stateless servers, restarts possible)

These constraints heavily influence failure modes.

3. What’s Actually Going Wrong (Root Causes)
❌ Problem 1: GPT-5 + Chat Completions is unstable / incorrect

You’re calling:

openai.chat.completions.create({
  model: "gpt-5",
  ...
})


Reality check:

GPT-5 is not guaranteed to support chat.completions

Structured JSON adherence is worse on reasoning-heavy models

GPT-5 prefers the Responses API, not Chat Completions

Symptoms you’ll see:

Empty message.content

Partial JSON

Extra prose

JSON wrapped in markdown

Random refusals to follow “EXACT format”

This alone explains most of your errors.

❌ Problem 2: You’re Overloading One Prompt With 4 Jobs

Your extraction prompt asks the model to:

Parse entire conversation history

Extract slots

Decide readiness

Generate user-facing dialogue

Obey strict JSON

Respect business rules

Respect delegation logic

Keep responses short

That’s too many competing objectives.

When models fail JSON, it’s almost always because:

“I tried to be helpful instead of precise.”

This is a design issue, not a prompt-wording issue.

❌ Problem 3: You’re Forcing Strict JSON Without Hard Guardrails

This line:

response_format: { type: "json_object" }


does NOT guarantee valid JSON.

It only nudges the model.

So when GPT-5 decides:

to clarify,

to rephrase,

to acknowledge,

to hedge,

your parser explodes.

Your fallback helps UX, but your state machine is now drifting.

❌ Problem 4: READY_TO_GENERATE Is Split Across Two Systems

You have:

A system-level instruction: “Respond with READY_TO_GENERATE…”

A JSON-level flag: readyToGenerate: true

These can—and will—disagree.

Example failure:

Model sets readyToGenerate: true

But conversational response doesn’t contain the prefix

Or vice versa

You now have desync between AI intent and backend state.

That’s a classic orchestration bug.

❌ Problem 5: Stateful Wizard + In-Memory Map = Silent Resets
wizardConversations.set(conversationId, ...)


This will break when:

The server restarts

Railway redeploys

Memory is reclaimed

Multiple instances spin up

You’ll get:

“Conversation not found”

Lost slots

Phantom readiness failures

This shows up as “AI is acting weird” but it’s infra-level.

4. Options & Trade-offs
Option A — Patch It (Least Work, Still Fragile)

Switch GPT-5 → GPT-4.1

Keep Chat Completions

Loosen JSON strictness

Accept occasional wizard weirdness

Trade-off: still brittle under scale

Option B — Proper AI Architecture (Recommended)

Split the flow into three deterministic steps:

1️⃣ Conversational Model (Freeform)

No JSON

No extraction

Just ask the next best question

GPT-4.1 or GPT-4o-mini

2️⃣ Slot Extractor (Hidden, Deterministic)

Short input

JSON-only

Small model

One job only

3️⃣ Readiness Logic (Your Code, Not AI)
const ready =
  campaignType &&
  targetAudience &&
  goals &&
  emailCount;


No AI judgment here.

This is how production CRMs do it.

Option C — Use the Responses API Properly (Advanced)

If you insist on GPT-5:

Use client.responses.create

Define a JSON schema

Use tool-style output

No mixed prose

Trade-off: more refactor, but clean.

5. Recommendation (Strong)

Do NOT use GPT-5 for this wizard right now.

Your use case requires:

High determinism

Predictable formatting

Low creativity

That is anti-GPT-5 territory.

Do this instead:

Conversation: gpt-4.1

Extraction: gpt-4.1-mini (JSON-only)

Generation: GPT-5 (where creativity helps)

This alone will remove ~80% of your errors.

6. Immediate Next Actions (Priority Order)

Switch GPT-5 → GPT-4.1 in /message

Split:

dialogue

extraction

readiness logic

Remove READY_TO_GENERATE from AI output entirely

Compute readiness in backend code

Persist conversations in DB or Redis (not memory)

Add JSON schema validation before parsing

7. Risks If You Don’t Fix This

Campaigns generated with missing slots

Phantom readiness triggers

Users stuck in loops

Silent failures after deploys

Impossible-to-debug “AI randomness”

This will absolutely surface once you pass ~20–30 concurrent users.
